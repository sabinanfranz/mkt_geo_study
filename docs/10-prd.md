# 10 — PRD (Product Requirements Document)

## 문제 정의

패스트캠퍼스 B2B 마케팅팀이 **GEO(Generative Engine Optimization)**라는 새로운 마케팅 분야를 학습할 때 다음 어려움을 겪는다:

1. **개념의 복잡성**: SEO와의 차이, RAG 5단계, 5대 신호 등 추상적 개념이 많음
2. **실무 거리감**: 이론 학습 후 실제 적용이 불명확함
3. **피드백 부족**: 자유롭게 사고하되 방향성 있는 가이드가 필요
4. **학습 동기 저하**: 일방적 강의보다는 interactive한 경험 필요

---

## 대상 사용자

- **주 사용자**: 패스트캠퍼스 B2B 마케팅팀 (20~50명)
- **역할**: 마케팅 담당자, 마케팅 매니저 (기본 디지털 마케팅 경험 있음)
- **경험 수준**: GEO 초보자 (0~1개월 경험)
- **학습 목표**: GEO 기초 개념 이해 → 간단한 기술적 실습

---

## 핵심 가치 제안

### 학습자 관점
- **단계별 가이드**: "전문가가 이렇게 생각했어요" 방식의 선택지 기반 멘토링
- **실무 연결성**: 이론(Stage 1)과 기초 실습(Stage 2)의 명확한 단계
- **즉각적 피드백**: 각 선택 후 사전 작성된 피드백으로 왜 그렇게 선택하는지 이해

### 조직 관점
- **콘텐츠 통제**: 자유 텍스트 입력 없이 일관된 메시지 유지
- **재활용성**: 종합평가 결과로 학습자 레벨 파악 → 추후 advanced 과정 설계에 활용
- **측정 가능**: 각 선택 기록 → 사용자별 취약점 분석 가능

---

## 범위 (In-Scope)

### 학습 콘텐츠
- **Stage 1 "GEO 세계관 이해"** (8개 모듈)
  - M1-1: GEO 정의
  - M1-2: SEO vs GEO
  - M1-3: RAG 5단계
  - M1-4: 5대 신호
  - M1-5: 3층 KPI
  - M1-6: 근거 vs 가설
  - M1-7: 전문가 분석 시연
  - M1-8: 종합평가 (70% 이상 달성 시 Stage 2 해제)

- **Stage 2 "Technical GEO 기초 실습"** (8개 모듈)
  - M2-1: IA(Information Architecture) 개념
  - M2-2: 허브-클러스터 맵
  - M2-3: Answer-first 원칙
  - M2-4: FAQ 설계 실습
  - M2-5: 헤딩 재설계
  - M2-6: 내부 링크 최적화
  - M2-7: 80/20 판단 원칙
  - M2-8: 종합평가

### 인터랙션 패턴
- **패턴 A (읽기 카드)**: 마크다운 콘텐츠 + "다음" 버튼
- **패턴 B (선택 퀴즈)**: 4지선다 또는 O×X → 정답 확인 → 사전 작성된 피드백
- **패턴 C (다단계 실습)**: 2~3단계 선택 과정 → 조합 결과 → 전문가 분석과 비교

### 기술 기능
- **대시보드**: Stage별 진도 바 + 현재 위치 + 다음 학습 권유
- **모듈 목록**: Stage별 8개 모듈 표시 (완료 / 진행중 / 잠금)
- **학습 화면**: Step별 콘텐츠 로드 → 선택지 제시 → 피드백 → "다음" or "완료"
- **결과 화면**: 종합평가 답안 선택 → 정답율 표시 → 새 Stage 해제 (또는 재시도 권유)
- **진도 관리**: 사용자별 선택 기록 → 진도 추적 (단일 사용자 default)

### API (8개)
- `GET /api/health`: 서버 상태 확인
- `GET /api/stages`: 전체 Stage 목록 + 진도 상태
- `GET /api/stages/{id}/modules`: 특정 Stage의 모듈 목록
- `GET /api/modules/{id}`: 모듈 상세 정보
- `GET /api/modules/{id}/steps`: 모듈 내 Step 목록 + 콘텐츠
- `POST /api/steps/{id}/answer`: 선택지 제출 → 피드백 반환
- `GET /api/progress`: 사용자 진도 조회
- `POST /api/stages/{id}/evaluate`: 종합평가 점수 계산 → Stage 해제 판정

### 데이터베이스 (5개 테이블)
- `stages`: Stage 메타데이터 (id, title, description, unlock_condition)
- `modules`: 모듈 메타데이터 (id, stage_id, title, order)
- `steps`: Step 정의 (id, module_id, type, content, options, correct_answer, feedback)
- `options`: 선택지 (id, step_id, text, is_correct, feedback)
- `user_progress`: 사용자 진도 (id, stage_id, module_id, step_id, selected_option_id, timestamp)

### 기술 스택
- **Frontend**: HTML / CSS / JavaScript (바닐라, 경량 라이브러리 최소화)
- **Backend**: FastAPI (Python)
- **Database**: SQLite
- **콘텐츠 소스**: `materials/` 폴더 내 3개 문서에서 큐레이션

---

## 비범위 (Out-of-Scope)

- **LLM 기반 피드백**: 모든 피드백은 사전 작성된 텍스트 (LangGraph 미사용)
- **자유 텍스트 입력**: 모든 입력은 선택지 기반 (open-ended 질문 없음)
- **사용자 인증**: 단일 사용자(default) 진도 관리로 시작 (로그인 불필요)
- **다국어**: 한국어만 지원
- **모바일 최적화**: 데스크톱 웹 우선 (반응형은 추후)
- **실시간 협업**: 개인 학습용 (그룹 학습 기능 없음)
- **Stage 3 이상**: MVP는 Stage 1~2 (총 16개 모듈)
- **고급 분석**: 사용자 학습 분석 대시보드 (관리자 기능 제외)
- **콘텐츠 관리 UI**: Seed 데이터는 JSON 파일로 수동 관리 (CMS 구축 안 함)

## 성공 기준

| 지표 | 목표 | 측정 방법 |
|------|------|----------|
| **학습 완료율** | Stage 1 사용자 80% 이상이 M1-8(종합평가) 완료 | DB user_progress 집계 |
| **Stage 2 진입율** | Stage 1 완료자 중 70% 이상이 Stage 2로 진입 | unlock_condition 적용 여부 |
| **선택 정확도** | Stage 1 M1-1~M1-7 평균 정답율 60% 이상 | 각 step의 is_correct 기록 |
| **API 응답 시간** | 모든 GET 요청 < 500ms, POST 요청 < 1000ms | 서버 로그 분석 |
| **UX 완성도** | 3개 인터랙션 패턴(A/B/C) 모두 정상 작동 | 수동 테스트 체크리스트 |
| **콘텐츠 포함도** | 16개 모듈 × 평균 5개 Step = 80개 이상 Step 콘텐츠 완성 | Step 테이블 레코드 수 |

## 리스크 및 완화 방안

| 리스크 | 영향 | 확률 | 대응 |
|--------|------|------|------|
| **콘텐츠 큐레이션 지연** | 선택지 작성 시간 부족 → Seed 데이터 불완전 | 중 | MVP 16개 모듈만 먼저 진행, 콘텐츠 담당자와 병렬 진행 |
| **Stage 1 70% 난이도 미정합** | 너무 쉬우면 의미 없음 / 너무 어려우면 좌절 | 중 | 초기 3~5명 베타 테스트 → 정답율 분포 확인 후 조정 |
| **선택지 기반 학습의 한계** | 사용자가 "왜 이 선택이 정답인가"를 아직도 이해 못 할 수 있음 | 중 | 피드백 텍스트에 "핵심 이유" 명시 + 확장 미션 카드로 탐구 기회 제공 |
| **DB 성능 저하** | 사용자 100명 × 80개 Step → user_progress 테이블 대규모화 | 낮음 | SQLite 인덱싱 최적화 (user_id, module_id 복합 인덱스) |
| **모바일 접근성 부족** | 일부 사용자가 모바일로 접근 시도 | 중 | 대시보드에 "데스크톱 권장" 배너 표시 |
| **Stage 2 해제 후 이탈** | Stage 2 콘텐츠 부족 시 사용자 중단 | 낮음 | Stage 2 Seed 데이터를 Slice 2 완료 전에 준비 (병렬 진행) |

## 추가 고려사항

### 인지적 도제 이론 구현
MVP는 Cognitive Apprenticeship의 6원리를 선택지 기반으로 구현:
1. **모델링**: 전문가 분석 과정을 Step으로 관찰
2. **코칭**: 각 선택지에 피드백 제공
3. **스캐폴딩**: 초반 Step은 힌트 O, 후반 Step은 힌트 X (모듈별 난이도 상승)
4. **명료화**: "이유를 골라주세요" 형태의 2차 선택지 (패턴 B의 피드백 선택)
5. **반성**: 종합평가 후 나의 답 ↔ 전문가 답안 비교 UI
6. **탐구**: 확장 미션 카드 (선택적, P1 이상)

### Stage 해제 정책
- **Stage 1-8 (종합평가)**: 정답율 70% 이상 달성 → Stage 2 자동 해제
- **Stage 2-8 종료**: MVP 범위 (Stage 3+ 미정)

---

## 정의

- **선택지 기반**: 사용자가 제시된 옵션 중 하나를 클릭해서만 진행
- **Seed 데이터**: MVP 시작 시점에 필요한 모든 콘텐츠를 JSON/SQL로 미리 준비
- **단일 사용자**: 현재 사용자는 1명 (default), 추후 다중 사용자 지원 예정
- **종합평가**: 모듈 마지막 Step으로, 학습 효과 측정 + Stage 해제 판정용
