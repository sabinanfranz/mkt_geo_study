## 1) TL;DR (초압축)

### 전략 한 줄
`b2b.fastcampus.co.kr`를 AI 검색(ChatGPT Search + Google AI Overviews/AI Mode)에서 더 자주 인용되게 만들기 위해, **상단 정답문장 + 근거블록 + 팬아웃 콘텐츠 + 측정 루프**를 90일간 반복 실행한다.

### 가장 중요한 4개 실행 레버
1. **Answer-first**: 페이지 상단 2~6문장으로 정의/대상/제공형태/산출물/도입조건을 고정  
2. **Proof-first**: 통계/인용/외부 출처(3~7개) 블록을 Answer 바로 아래에 표준화  
3. **Fan-out 클러스터**: 허브 5~8개 + 스포크 50+로 하위 질의 커버리지 확대  
4. **측정 실험 루프**: Top 50 질문 세트로 주간 인용률/커버리지 추적 및 4주 단위 실험

### 30/60/90일 목표
- **30일**: 상위 랜딩 구조화 + 크롤러/측정 기본 세팅
- **60일**: 허브/스포크 대량 확장 + 사례/리포트 자산화
- **90일**: 엔티티/구조화데이터 고정 + 외부 권위 링크 + 실험 자동화

## 2) 소스에서 확정된 사실 vs 가설

## 2.1 확정 사실 (원문에서 명시)
- Google AI 기능은 “특별한 추가 기술요건”보다 기존 SEO 기본기(인덱싱/스니펫 가능)가 중요하다는 원칙을 따른다.
- Google AI Mode/Overviews는 질의를 하위 토픽으로 분해하는 fan-out 성격을 가진다.
- ChatGPT Search는 OAI-SearchBot 관련 제어/추적(예: `utm_source=chatgpt.com`)이 중요한 운영 포인트다.
- 현재 사이트에는 서비스/사례/리소스 자산이 풍부하고, 사례 페이지는 “대상/시수/방식/만족도” 같은 정형 필드 잠재력이 높다.
- 우선순위는 기술 미세조정보다 콘텐츠 구조(정답 문장/근거)와 측정 체계 정착이 먼저다.

## 2.2 가설/검증 필요 항목
- robots/sitemap/canonical/hreflang/pagination의 실제 운영 상태는 별도 실사 필요
- 일부 페이지 렌더링 상태(특히 정책/신뢰 페이지)의 크롤러 가독성은 추가 점검 필요
- 외부 권위 링크가 AI 인용률을 얼마나 직접적으로 올리는지는 실험으로 검증 필요

운영 원칙:
- “가설”로 분류한 항목은 즉시 백로그 티켓으로 분리하고, 인프라·SEO 점검 결과를 근거로 재분류한다.

## 3) 채널 메커니즘 요약 (ChatGPT vs Gemini)

| 항목 | ChatGPT Search | Google AI Overviews / AI Mode |
|---|---|---|
| 검색 동작 | OAI-SearchBot 기반 + 파트너 검색 결합 가능 | Google Search 인덱스 기반 |
| 핵심 관점 | 크롤 허용/차단 정책, 유입 추적(UTM) 명확 | 인덱싱·스니펫 가능성 + 웹검색 통합 리포트 |
| 팬아웃 | 질의 재작성/세분화 특성 존재 | query fan-out 명시적 특성 |
| 운영 리스크 | 크롤러 차단/정책 충돌 시 인용 급감 가능 | AI 기능 트래픽 분리 측정 한계 |
| 측정 포인트 | `utm_source=chatgpt.com` 기반 유입/전환 | Search Console(Web) + 수동 SERP 관찰 병행 |

의사결정 요점:
- 둘 다 결국 **“인용 가능한 텍스트 구조 + 신뢰 근거 + 크롤 가능성”**이 공통 분모다.
- 채널별로 다른 것은 “측정 방법”과 “통제 포인트”다.

## 4) 사이트 베이스라인 진단 압축

## 4.1 현재 강점
- 서비스/사례/리소스 축이 이미 존재 (콘텐츠 확장 기반 좋음)
- 사례 페이지의 정형 데이터(대상/시수/만족도)는 인용 친화적 자산으로 전환 가능
- 리포트/세미나/비즈레터 허브가 있어 팬아웃 클러스터로 확장 용이

## 4.2 현재 약점
- 상단에 즉답형 “정답 문장” 부재 페이지가 다수
- 정량/출처 근거 블록이 표준화되지 않음
- 페이지별 Trust 신호(작성/검수/업데이트 정책) 일관성 낮음
- 기술 신호(robots/canonical 등) 일부는 검증 미완

## 4.3 상위 개선 타깃 (1차)
1. `/service_custom`
2. `/service_online` / `/service_online_enterprise`
3. `/refer_customer` + 주요 사례 상세
4. `/resource_report` / `/resource_seminar` / `/resource_bizletter`
5. `/contact_mktg`

선정 이유:
- 전환 기여, 허브성, 확장성, 정형화 잠재력 기준

## 5) 전략 설계 핵심 (Content System + Technical + Authority + Measurement)

## 5.1 GEO Content System 표준

### A. Answer-first 표준
- 위치: H1 직하단
- 길이: 2~6문장
- 필수 항목 순서:
  - 정의
  - 대상
  - 제공 형태
  - 산출물
  - 도입 조건/리드타임

### B. Proof-first 표준
- 위치: Answer-first 바로 아래
- 구성:
  - 통계 1~3개 (기간/표본/정의 포함)
  - 1차 인용문 1개
  - 외부 출처 링크 3~7개
- 원칙: 페이지에서 보이는 내용과 일치해야 함

### C. Fan-out 클러스터 표준
- 허브 5~8개
- 스포크 50~150개
- 스포크 포맷(600~1,200자):
  - 1문단 요약
  - 절차
  - 체크리스트
  - FAQ
  - Proof 블록

## 5.2 Technical & Structured Data 우선순위

### 1순위
- Organization
- BreadcrumbList

### 2순위(조건부)
- Event (세미나/일정형 페이지)
- Article (인사이트/리포트)
- Course (적용 범위/효과 사전 검증 후)

### robots/noindex 운영 원칙
- 검색 인용 허용과 학습 차단을 분리해 정책 설계 가능
- noindex는 크롤러가 읽을 수 있어야 유효하므로 “크롤 허용 + 메타 noindex” 설계 주의

## 8) 다른 LLM에게 넘길 때의 작업 규칙

## 8.1 입력 프롬프트 규칙
- “무엇을 만들지”보다 “어떤 KPI를 올릴지”를 먼저 명시
- 페이지 단위 작업은 항상 아래 5요소를 함께 요청
  - 대상 URL
  - 사용자 의도
  - Answer-first 초안
  - Proof 블록 초안
  - 측정 항목

## 8.2 산출물 포맷 규칙
- 모든 제안은 `Impact / Effort / Risk / DoD / KPI` 5열 표로 반환
- 카피 제안은 “상단 2~6문장”과 “근거 블록”을 분리해서 작성
- 실험 제안은 “가설-실험군-대조군-기간-판정기준”을 반드시 포함

## 8.3 금지/주의
- 막연한 브랜딩 문구만 제안 금지 (정의/스펙 없는 카피 금지)
- 출처 불명 통계 사용 금지
- robots와 noindex 충돌을 무시한 제안 금지

## 10) 최종 요약

이 전략의 본질은 “기술 트릭”이 아니라 **인용 가능한 정보 구조를 운영 시스템으로 만드는 것**이다.  
즉, 페이지를 예쁘게 바꾸는 것이 아니라, AI가 재사용하기 쉬운 정답 단위(Answer)와 검증 단위(Proof)를 반복 생산하고, fan-out 질문망에서 측정 가능한 방식으로 커버리지를 확장하는 것이 핵심이다.

LLM이 이 문서를 읽고 해야 할 첫 일:
1. 상위 10 랜딩에 대한 Answer-first/Proof-first 초안 생성
2. Top 50 질문과 매핑한 4주 실험표 작성
3. P0 티켓 5개를 주간 실행 backlog로 전개
